# Conan Exiles | Server Manager Enhanced — v0 design

**Date:** 2026-05-08
**Status:** Validated, ready to scaffold
**Owner:** Brandon Bridges

A free, open-source desktop control panel for Conan Exiles Enhanced (UE5) dedicated servers. v0 ships the **live-admin wedge**: the things RCON exposes that the existing tooling doesn't surface in a modern UI.

---

## Context

Conan Exiles Enhanced shipped on 2026-05-05 — Funcom's free UE5 upgrade of the base game. The community is actively reconfiguring servers, mod authors are republishing, hosting providers are scrambling to support the new build. The dominant existing tool, *Conan Mod and Server Manager* on Nexus, is Windows-only, FTP-based, ageing WinForms, and offers no live admin.

There is a 4–6 week window where being the new shiny thing matters. v0 is timed to ship inside that window with a sharp, opinionated scope.

**Commercial target:** £300–500/month recurring within 12 months, achieved via a paid Pro hosted sidecar (v2.0). v0 itself is free and open-source — it builds the audience that Pro sells to.

## Stack

- **Tauri 2.x** (Rust core + webview UI) over Electron. ~10 MB single static binary, native OS keychain access via `keyring` crate, faster cold start, no Node runtime baked in.
- **Vite + React + TypeScript** for the UI. Not Next.js — no SSR or server routing needed in a desktop shell.
- **shadcn/ui** + Tailwind for components. Already familiar from Orbit.
- **SQLite** via `rusqlite` for local persistence. Connection metadata only.
- **Rust crate `rcon`** for the BattlEye RCON wire protocol. Wrapped in our own `packages/rcon-client` for the connection-pool and reconnect logic.
- **`ts-rs`** to generate TypeScript types from Rust structs — single source of truth for the cross-language contract.

## Targets

- Windows x64
- macOS (universal: Apple Silicon + Intel)
- Linux deferred to v0.1

## Architecture (three layers)

1. **Rust core (`src-tauri/`)** — RCON connection pool, SQLite I/O, keychain integration, polling scheduler, Tauri command handlers exposed to the webview.
2. **Webview UI (`src/`)** — React app, talks to Rust via Tauri `invoke()` and event subscriptions.
3. **Shared types** — generated from Rust via `ts-rs`. Build step regenerates `src/types/generated.ts` on every Rust change.

## Credential storage

- **OS keychain** for all secrets (RCON password, optional AdminPassword). Two entries per server: `cesm:{server_id}:rcon` and `cesm:{server_id}:admin`. macOS Keychain on darwin, Windows Credential Manager on win32.
- **SQLite** holds connection metadata only (name, host, port, has_admin_pw flag, timestamps). At `$APP_DATA/cesm/db.sqlite`.
- Passwords never written to disk. Never logged.

## Screens

Six views (the connection form is a modal — five proper pages):

1. **Onboarding / empty state** — first launch shows a centred "Add your first server" CTA. Functional only.
2. **Sidebar (persistent shell)** — left rail. Saved servers with status dot (green/grey/red), "Add server" button at top, settings cog at bottom. Discord-style.
3. **Connection form (modal)** — Add/Edit. Fields: display name, host/IP, RCON port (default 25575), RCON password, optional AdminPassword. "Test connection" before save. Passwords write straight to keychain.
4. **Dashboard (per server)** — top hero: status pill, current player count, uptime, server name from `serverinfo`. Below: broadcast composer (textarea + send), live event feed (join/leave events from `listplayers` polling diff). One-click `Save world` and `Restart` gated behind confirmation.
5. **Players (per server)** — table of currently online players. Columns: name, FuncomID/SteamID, ping, time on server, actions. Per-row buttons: **Kick** (reason input), **Ban** (reason + optional duration), **Promote to admin**. Sub-tab "Bans" lists cached banned players with **Unban**. Search/filter at top.
6. **Console (per server)** — raw RCON. Command input with history (↑/↓), scrollback, syntax-highlighted. Power-user escape hatch.
7. **Settings (app)** — theme toggle, default polling interval (10s default, range 5–60s), update channel selector (Stable/Beta), about/version, "Open data folder" link.

## RCON command surface (v0)

Buttons map to these commands:

- `listplayers` — used for player list polling
- `kickplayer <name> [reason]`
- `banplayer <name> [reason]`
- `unbanplayer <name>`
- `listbans` / `banlist`
- `broadcast <message>`
- `serverinfo`
- `saveworld`
- `whitelistplayer` / `unwhitelistplayer` (exposed in console only for v0)

**Admin promotion** — Conan's RCON has no "elevate this player" command. Self-elevation via `/MakeAdmin <password>` in chat is the only path. v0 implementation:

- Connection setup optionally collects the `AdminPassword`.
- "Promote to admin" button on a player row sends a private broadcast: "You've been granted admin. Type `/MakeAdmin <password>` in chat to activate."
- "Demote" sends a prompt to type `MakeMeNormal`.
- Honest UI copy explains it's a prompt, not telepathic. Deeper integration (e.g. AdminPassword rotation via `SetServerSetting` to invalidate) deferred to v1.

Console view covers anything not exposed as a button, including future commands Funcom adds.

## Data model

Local SQLite schema, three tables:

```sql
CREATE TABLE servers (
  id            TEXT PRIMARY KEY,    -- uuid v4
  name          TEXT NOT NULL,
  host          TEXT NOT NULL,
  rcon_port     INTEGER NOT NULL,
  has_admin_pw  BOOLEAN NOT NULL,
  created_at    INTEGER NOT NULL,
  last_used_at  INTEGER
);

CREATE TABLE ban_cache (
  server_id     TEXT NOT NULL,
  player_id     TEXT NOT NULL,
  player_name   TEXT,
  reason        TEXT,
  banned_at     INTEGER,
  PRIMARY KEY (server_id, player_id),
  FOREIGN KEY (server_id) REFERENCES servers(id) ON DELETE CASCADE
);

CREATE TABLE app_settings (
  key   TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

No remote sync. No accounts. Local-only.

## RCON connection lifecycle

- Open lazily when a server is selected.
- Keep alive while the server is the active view.
- Close on switch or app blur.
- Single connection per server (RCON is single-session).
- Reconnect with exponential backoff on drops (1s, 2s, 4s, 8s, max 30s).

## Polling

- `listplayers` every 10s while the server is the active view (configurable 5–60s in Settings).
- Diff against previous snapshot to derive join/leave events.
- Pause polling when the OS window is unfocused — saves bandwidth, avoids spamming the server.
- Event feed is an in-memory ring buffer of last 200 events per server. Clears on app restart. Persistence is a v1 concern.

## Release channels

Two channels, single development branch. Differentiated by tag pattern, not branch.

- **Stable** — `vX.Y.Z` tags. Cut by release-please when its release PR is merged. Default channel for new installs. Code-signed, attached to GitHub Releases. Auto-update prompts.
- **Beta** — `vX.Y.Z-beta.N` tags. Cut manually from `main` when a feature batch wants wider testing. Marked as pre-release on GitHub. Opt-in via Settings.

**Tauri updater serves two manifests:**

- `updater/stable.json`
- `updater/beta.json`

Each regenerated by its workflow when a release in that channel publishes. Desktop app polls the manifest matching the user's chosen channel.

**Settings → Update channel:**

- **Stable** (default) — "Solid, tested releases."
- **Beta** — "Early access to upcoming features. Occasional bugs."

App titlebar/About shows the current channel and version. Beta users see a "Report a bug" link pre-filled with diagnostic info.

**Workflows:**

- `release.yml` — handles stable releases.
- `beta.yml` — manual dispatch + run on push of any `v*-beta.*` tag. Build, sign, regenerate `beta.json`.

**Contributor recognition:** beta users who file accepted bug reports get credited in `CONTRIBUTORS.md`. Future Discord adds a `Tester` role.

## Open questions / risks

- **Conan RCON command verification** — Funcom occasionally renames/changes RCON commands between releases. The Enhanced launch is fresh; community wikis lag. The Week 1 spike must verify every command we wire to a button against a real Enhanced server before relying on it.
- **BattlEye RCON quirks** — Conan uses BattlEye RCON, which differs subtly from Source RCON. The `rcon` crate may need patching or we write a thin adapter in `packages/rcon-client`. Spike Week 1.
- **Tauri 2 macOS code-signing** — process changed in Tauri 2.x; first signed build will eat half a day setting up notarisation. Budget for it in Week 3.
- **Existing tool incumbency** — *Conan Mod and Server Manager* has years of installed base. We win on UX + cross-platform + live admin, but adoption isn't guaranteed. Nexus listing copy and launch posts must lead with the differentiators, not duplicate the existing tool's pitch.
- **Apple Developer cert delay** — first-time Apple Developer enrolment can take days to a week. Apply Week 1, not Week 3.
- **Conan player base size** — smaller than 7 Days to Die or Ark. £300/mo target requires ~60 paying servers at v2 launch. Plausible but tight; mod-heavy servers are the most likely converters.
